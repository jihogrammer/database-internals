Chapter 5: Transaction Processing and Recovery

Topics: ACID Page cache WAL ARIES

This chapter moves from low-level storage to the components required for transactions: buffer management, locks, logging, and recovery. We explain transactions as logical units of work and the guarantees they must provide (ACID), then describe page caching, eviction policies, write-ahead logging, recovery techniques (including ARIES), and concurrency-control strategies.

ACID

A transaction is an indivisible logical unit of work that groups multiple operations (reads and writes). Transactions must preserve Atomicity, Consistency, Isolation, and Durability (ACID).

Atomicity

Either all steps of a transaction succeed, or none do. A committed transaction makes changes visible; an aborted transaction must roll back all side effects. Commit is final; after abort a transaction can be retried.

Consistency

Consistency is application-specific: a transaction should move the database from one valid state to another (respecting constraints, referential integrity, etc.).

Isolation

Concurrent transactions should behave as if executed serially. Many systems relax isolation for performance and provide isolation levels (see Isolation Levels).

Durability

Once committed, transaction effects must survive crashes and power loss; durability implies writing sufficient state to persistent storage.

Key Components

Implementing transactions requires coordinated components on the node: transaction manager, lock manager, page cache (buffer pool), and log manager.

The lock manager grants shared/exclusive locks to protect resources; the page cache stages changes in memory; the log manager keeps an append-only history (the WAL) used for replay and undo during recovery.

Buffer Management & Page Cache

Databases use a two-level memory hierarchy: persistent storage (disk) and main memory (RAM). The page cache (also called buffer pool) keeps disk pages in memory to reduce I/O.

Cached pages are referenced by the storage engine, can be pinned to avoid eviction, and when modified are marked dirty until flushed to disk. In a crash, cached-only changes are lost unless protected by the WAL.

Bypassing the kernel page cache

Many databases use O_DIRECT to bypass the kernel page cache and implement their own buffer management. This trades off kernel services (readahead, async) for application control. Other options include posix_fadvise and memory mapping.

Caching semantics

The page cache lets the engine decouple logical writes from physical writes. When a page is requested, the cache returns an in-memory copy or loads it from disk; modified pages are dirty until flushed. Pinning prevents eviction while in use.

Cache Eviction & Page Replacement

When memory fills, the cache must evict pages. Clean (in-sync) pages can be evicted immediately; dirty pages must be flushed first. Checkpointing coordinates flushes with the WAL to preserve durability.

Eviction policy matters. Naïve strategies can suffer Bélády’s anomaly where larger caches increase evictions.

FIFO and LRU

FIFO evicts by insertion order; LRU evicts least-recently used pages and updates access order on each access. LRU can be costly to maintain with high concurrency.

CLOCK

The CLOCK algorithm is a compact approximation of LRU using an access bit and a circular buffer; it is efficient and commonly used (Linux employs a variant).

LFU / TinyLFU

Frequency-based eviction (LFU) tracks access counts. TinyLFU combines a frequency filter with admission and protected/probation queues to favor frequently used items (used by Caffeine library).

Recovery & Write-Ahead Log (WAL)

Systems must survive failures at multiple layers. The write-ahead log (WAL) is an append-only structure recording operations so that in-memory changes can be reconstructed after a crash. The WAL is central to durability: an operation that modifies state must be logged before its pages are persisted.

PostgreSQL and fsync()

PostgreSQL uses checkpoints and fsync() to sync dirty pages. However, fsync semantics on some OSes can clear dirty flags even on I/O errors, potentially masking problems. Careful testing of failure scenarios is critical for recovery correctness.

Log semantics

WAL records have monotonically increasing log sequence numbers (LSNs). Log buffers are flushed (forced) to disk in LSN order. A transaction is not committed until its commit record is forced to disk. Compensation log records (CLRs) can be used during undo to make recovery robust.

Checkpoints

Checkpoints mark points where log records up to an LSN are durable and can be discarded. Fuzzy checkpoints avoid pausing the system: begin/end checkpoint records and async page flushes let recovery start from the last successful checkpoint.

Operation vs Data Log

Logs can be physical (page images) or logical (operations). Physical logging stores before/after images; logical logging stores operations. Many systems mix approaches (logical undo, physical redo) to balance concurrency and recovery speed.

Steal and Force policies

Steal/no-steal and force/noforce policies define when in-memory changes are flushed. Steal allows flushing uncommitted changes (requiring undo records); force requires flushing a transaction's pages before commit (slower). ARIES is a steal/no-force algorithm.

ARIES

ARIES (Algorithm for Recovery and Isolation Exploiting Semantics) uses WAL, physical redo and logical undo, and compensation log records. Recovery runs in three phases: analysis (identify dirty pages and in-flight transactions), redo (replay history up to crash), and undo (roll back incomplete transactions).

Concurrency Control

Concurrency control handles interactions among concurrent transactions. Approaches include optimistic concurrency control (OCC), multiversion concurrency control (MVCC), pessimistic (lock-based) schemes, and timestamp ordering.

Serializability

A schedule is serializable if it is equivalent to some serial execution of the same transactions. Serializable schedules allow concurrent execution while preserving the correctness of a serial schedule.

Isolation Levels & Anomalies

SQL defines anomalies: dirty reads, nonrepeatable reads, and phantom reads. Isolation levels (read uncommitted, read committed, repeatable read, serializable) trade performance for stronger guarantees. Snapshot isolation provides a snapshot of committed state at transaction start but can allow write skew.

Optimistic Concurrency Control (OCC)

OCC splits execution into read, validation, and write phases. Transactions execute without blocking, then validate that they do not conflict; conflicts cause abort and retry. Validation can be backward- or forward-oriented.

Multiversion Concurrency Control (MVCC)

MVCC keeps multiple timestamped versions of records so readers can see a consistent snapshot. It reduces coordination for reads and is often used to implement snapshot isolation.

Pessimistic & Lock-Based Control

Pessimistic schemes detect conflicts during execution and use locks to block or abort. Two-phase locking (2PL) is widespread: acquire locks in a growing phase and release them in a shrinking phase. Locks introduce contention and risk deadlocks.
