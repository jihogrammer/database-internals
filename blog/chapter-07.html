7장 - LSM-Tree: 쓰기 최적화와 불변성

작성일: 2026-03-22

개요

LSM-Tree(Log-Structured Merge Tree)는 B-Tree의 대안으로, 쓰기 성능을 극대화하기 위해 설계된 데이터 구조입니다. 불변성(Immutability)을 기반으로 하여 쓰기 작업을 순차적으로 처리하고, 배경에서 병합을 수행합니다.

배경: 하드웨어의 진화와 자연선택설

"회계사는 지우개를 쓰는 순간 철창행이다." — 팻 헬란드 (Pat Helland)

과거 HDD 시대에는 탐색 시간(Seek time) 때문에 랜덤 I/O를 줄이는 B-Tree의 구조가 지배적이었습니다. 하지만 SSD 시대가 도래하며 판도가 바뀌었습니다.

  • SSD의 특징: 랜덤 쓰기 시 '지우기 전 쓰기(Erase-before-write)' 제약으로 인해 성능 저하와 수명 단축이 발생합니다. 반면, 순차 쓰기는 압도적인 성능을 냅니다.
  • 소프트웨어의 응답: LSM-Tree는 모든 쓰기를 순차적 추가(Append-only)로 처리하여 SSD의 물리적 잠재력을 끌어냅니다. 이는 기술적 자연선택의 결과입니다.

핵심 개념

1. LSM-Tree의 구조

  • MemTable: 메모리에 있는 쓰기 버퍼. 정렬된 키-값 쌍을 저장합니다.
  • SSTable (Sorted String Table): 디스크에 있는 불변 파일. 정렬된 키-값 쌍을 포함합니다.
  • Compaction: 여러 SSTable을 병합하여 크기를 줄이고 읽기 성능을 개선합니다.

2. Memtable (메모리 테이블)

실시간 쓰기를 받아내는 인메모리 구조입니다. 일반적으로 Skip List나 Red-Black Tree를 사용하여 항상 정렬된 상태를 유지합니다. 크기 제한(예: 64MB)에 도달하면 Immutable Memtable로 전환되고, 새 Memtable이 생성되어 쓰기를 계속 받습니다.

3. WAL (Write-Ahead Log)

Memtable의 휘발성을 보완하기 위해 디스크에 기록하는 순차 로그입니다. 트랜잭션 커밋 시점에 WAL에 먼저 기록되며, 장애 발생 시 WAL을 재생하여 Memtable을 복구합니다. WAL은 SSTable로 플러시된 이후에는 안전하게 삭제됩니다.

4. SSTable (Sorted String Table)

Memtable이 가득 차서 디스크로 내려온 불변(Immutable) 파일입니다.

  • 내부 구조:
    • 데이터 블록(Data Block): 실제 키-값 쌍이 정렬되어 저장됩니다. (보통 4KB~64KB)
    • 인덱스 블록(Index Block): 각 데이터 블록의 마지막 키와 오프셋을 저장하여 빠른 위치 탐색을 돕습니다.
    • 블룸 필터(Bloom Filter): 해당 SSTable에 특정 키가 존재하는지 여부를 빠르게 판별합니다.
    • 메타데이터 블록: 필터, 통계 정보 등을 저장합니다.
  • 한 번 써진 파일은 절대 수정되지 않으므로, 업데이트나 삭제는 새로운 SSTable에 반영됩니다.

5. 쓰기 과정

  1. 쓰기 요청이 MemTable에 추가됩니다.
  2. MemTable이 가득 차면 SSTable로 플러시됩니다.
  3. 읽기 시 MemTable과 SSTable을 모두 검색합니다.

7. Compaction 세부 사항

Leveled Compaction (LevelDB, RocksDB 기본)

  • 데이터가 여러 레벨(Level 0 ~ Level N)로 구성됩니다.
  • Level 0: Memtable에서 플러시된 직후의 SSTable. 파일 간 키 범위가 겹칠 수 있습니다.
  • Level 1 이상: 각 레벨 내에서는 키 범위가 겹치지 않도록 유지합니다.
  • 컴팩션 과정: Level 0의 파일들이 특정 개수를 넘으면, 겹치는 Level 1 파일들과 병합하여 새로운 Level 1 파일을 생성합니다.
  • 특징: 읽기 성능이 우수하지만, 쓰기 증폭이 큽니다.

Size-Tiered Compaction (Cassandra, HBase 초기)

  • 동일한 크기의 SSTable이 일정 개수(예: 4개) 쌓이면, 이들을 병합하여 더 큰 하나의 SSTable을 만듭니다.
  • 특징: 구현이 단순하고, 쓰기 증폭이 낮지만, 읽기 시 여러 파일을 뒤져야 할 가능성이 커집니다.

알고리즘: K-way Merge

여러 개의 정렬된 SSTable을 우선순위 큐(Min-heap)를 사용하여 한 번의 순차 스캔으로 병합합니다.

Write Stall (쓰기 제동)

컴팩션 속도가 쓰기 속도를 못 따라가면 시스템은 스스로 쓰기 속도를 늦춥니다. 이는 시스템 붕괴를 막는 자기 보호 기법입니다.

8. Bloom Filter

읽기 성능을 높이기 위해 SSTable에 Bloom Filter를 추가합니다. Bloom Filter는 키가 존재하지 않을 가능성을 빠르게 확인하여 불필요한 디스크 접근을 줄입니다.

9. 희소 인덱스 (Sparse Index)

모든 키가 아닌 데이터 블록의 경계 키만 인덱싱하여 메모리 효율을 극대화합니다.

LSM-Tree의 장점

  • 높은 쓰기 성능: 순차 쓰기로 SSD에 최적화
  • 불변성: 데이터 수정 대신 새 버전 생성
  • 확장성: 대용량 데이터 처리에 적합

실무 적용

  • LevelDB/RocksDB: 키-값 저장소로, Facebook과 같은 대규모 시스템에서 사용
  • Cassandra: 분산 데이터베이스로, 고가용성과 쓰기 성능을 위해 LSM-Tree를 채택
  • Elasticsearch: 검색 엔진으로, Apache Lucene의 인버티드 인덱스가 LSM-Tree 구조를 사용하여 실시간 인덱싱과 검색을 지원. 로그 분석, 모니터링 등 쓰기 중심 워크로드에 적합

Elasticsearch와 LSM 트리의 관계

Elasticsearch(Lucene)는 검색 엔진이지만, 쓰기 최적화와 불변성이라는 LSM 트리의 핵심 원리를 적극적으로 채택했습니다.

Elasticsearch의 차이점

  • 실시간성: Memory Buffer의 데이터는 검색 결과에 포함되지 않습니다. Refresh 작업(기본 1초)이 실행되어야 새로운 세그먼트로 플러시되며, 그제서야 검색이 가능해집니다. 따라서 근실시간(NRT) 검색 엔진입니다.
  • 데이터 구조: 역색인(Inverted Index)을 생성하여 전문 검색에 최적화되어 있습니다.

실무 운영 인사이트

  • Refresh Interval 튜닝: 대량 적재 시 refresh 간격을 늘려 쓰기 성능 향상
  • Segment Merge 관리: I/O 집약적인 작업이므로 부하 시간대에 제한
  • Translog 관리: 내구성과 성능의 균형 조절

Original English Text

Chapter 7: LSM-Tree: Write Optimization and Immutability

In the previous chapters, we discussed B-Trees and their variants. B-Trees are optimized for read-heavy workloads and provide good performance for random reads. However, they suffer from poor write performance due to random I/O. LSM-Trees address this limitation by optimizing for write-heavy workloads.

LSM-Trees are based on the principle of immutability. Instead of modifying data in place, LSM-Trees create new versions of data structures. This approach allows for sequential writes, which are much faster on modern storage devices.

The core idea behind LSM-Trees is to buffer writes in memory and periodically flush them to disk in sorted order. Background processes then merge these sorted files to maintain efficient read performance.

LSM-Tree Components

  • MemTable: An in-memory data structure that buffers recent writes.
  • SSTable: Immutable, sorted files on disk.
  • Write-Ahead Log (WAL): Ensures durability for in-memory data.
  • Compaction: Merges SSTables to reduce read overhead.

Write Path

  1. Write to WAL for durability.
  2. Insert into MemTable.
  3. When MemTable is full, flush to SSTable.

Read Path

  1. Check MemTable.
  2. Search SSTables in order (newest first).
  3. Use bloom filters to skip irrelevant files.

Compaction Strategies

Compaction is crucial for maintaining performance. Different strategies include size-tiered and leveled compaction, each with trade-offs in write amplification and read performance.

Advantages

  • High write throughput due to sequential I/O.
  • Efficient for large datasets.
  • Simpler crash recovery.

Disadvantages

  • Higher read latency due to multiple file searches.
  • Increased space usage during compaction.
  • Complex tuning of compaction parameters.

Use Cases

LSM-Trees are ideal for write-heavy applications like time-series databases, logging systems, and analytics platforms.
