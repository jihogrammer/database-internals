분산 시스템의 복제와 일관성 모델

Chapter 11 • 작성일: 2026-03-28

**"왜 옆 동네 친구가 본 데이터와 내가 본 데이터가 다를까?"** 8장(네트워크), 9장(장애 감지), 10장(합의)을 지나 드디어 도착한 분산 시스템의 결론, **일관성 모델**을 파헤쳐 봅니다.

🚀 0. 들어가며: 분산 시스템의 가혹한 현실

단일 DB에서는 Write 후 Read 하면 항상 최신 값이 나옵니다. 하지만 분산 시스템은 다릅니다.

  • 네트워크(8장): 패킷은 유실되거나 순서가 뒤바뀝니다.
  • 장애 감지(9장): 노드가 죽은 건지, 그냥 느린 건지 알 수 없습니다.
  • 합의(10장): 과반수(Quorum)의 동의 없이는 무엇도 '진실'이 되지 못합니다.

이 모든 불확실성 속에서 "데이터를 어디까지 믿을 수 있는가?"를 정의한 규칙이 바로 일관성 모델입니다.

📊 1. 일관성 모델 스펙트럼 (Consistency Spectrum)

일관성은 '된다/안 된다'의 이분법이 아니라, 성능과 정확성 사이의 선택(Trade-off)입니다.

graph LR
    A[선형성 Linearizability] --- B[순차적 일관성 Sequential]
    B --- C[인과적 일관성 Causal]
    C --- D[최종적 일관성 Eventual]
    style A fill: #f96, stroke: #333, stroke-width: 2px
    style D fill: #69f, stroke: #333, stroke-width: 2px

❶ 선형성 (Linearizability) - "가장 강력한 보장"

정의: 모든 연산이 전역적으로 하나의 순서로 즉시 반영됨.

특징: 마치 단일 노드 DB를 쓰는 것 같은 직관성을 제공합니다.

단점: 네트워크 지연에 극도로 취약하며 가용성(Availability)이 떨어집니다.

❷ 인과적 일관성 (Causal Consistency) - "현실적인 타협안"

정의: '원인'과 '결과' 관계가 있는 연산들의 순서만 보장함.

예시: 질문글(원인)이 올라온 뒤에야 답변글(결과)이 보여야 합니다.

특징: 선형성보다 빠르며, 대부분의 서비스에 충분한 수준입니다.

❸ 최종적 일관성 (Eventual Consistency) - "가용성 올인"

정의: 새로운 업데이트가 없다면, 시간이 흐른 뒤 모든 복제본이 동일해짐.

특징: 쓰기 직후 읽기는 옛날 값을 볼 수 있지만, 시스템은 멈추지 않고 돌아갑니다.

🛠 2. 실무 사례: 실제 DB는 어떻게 쓸까?

이론을 넘어 실제 시스템에서 일관성을 제어하는 방법입니다.

시스템 기본 모델 선형성(Strong) 확보 방법 사례
Cassandra 최종적 QUORUM 옵션으로 읽기/쓰기 강제 대규모 로그, 타임라인
Kafka 인과적/순차적 acks=all 설정으로 모든 복제 완료 확인 이벤트 스트리밍
DynamoDB 최종적 ConsistentRead=True 옵션 사용 장바구니, 주문

💡 주목! Redis의 WAIT 예시

SET user_1 "Gildong"
WAIT 1 5000  # 최소 1개 복제본에 복제될 때까지 5초간 대기

비동기 복제가 기본인 Redis에서도 WAIT를 통해 선형성에 가까운 보장을 얻을 수 있습니다.

⚖️ 3. CAP에서 PACELC로: 현실적인 선택

P(Partition) 상황: C(일관성)를 챙길 것인가, A(가용성)를 챙길 것인가?

E(Else, 정상) 상황: L(지연 시간)을 줄일 것인가, C(일관성)를 높일 것인가?

결론: "장애가 없을 때조차 우리는 성능(Latency)을 위해 일관성을 포기하곤 한다."우리는 흔히 CAP 이론(Consistency, Availability, Partition Tolerance)을 이야기합니다. 하지만 현실은 더 디테일합니다.

📝 4. 핵심 요약 및 퀴즈 (Self-Check)

핵심 요약

일관성은 이진적 선택이 아닌 스펙트럼이다.

선형성은 정확하지만 느리고, 최종적 일관성은 빠르지만 부정확할 수 있다.

쿼럼(Quorum) 설정을 통해 개발자가 직접 일관성 수준을 조정할 수 있다.
