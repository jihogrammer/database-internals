분산 시스템의 복제와 일관성 모델 완전 정리

Chapter 11 • 작성일: 2026-03-28
# 분산 시스템의 복제와 일관성 모델 완전 정리 > **"왜 옆 동네 친구가 본 데이터와 내가 본 데이터가 다를까?"** > > 8장(네트워크), 9장(장애 감지), 10장(합의)을 지나 드디어 도착한 분산 시스템의 핵심, **일관성 모델**을 파헤쳐 봅니다. --- ## 들어가며: 분산 시스템의 가혹한 현실 데이터베이스에서 복제(Replication)는 가용성, 확장성, 지리적 분산을 위해 필수적입니다. 하지만 복제본이 여러 개 존재하면 **일관성(Consistency)** 문제가 발생합니다. 단일 DB에서는 간단합니다: ``` write → read = 항상 최신 값 ``` 하지만 분산 시스템은 다릅니다: ``` Node A: X = 1 Node B: X = 0 ``` 이 모든 불확실성 속에서 **"데이터를 어디까지 믿을 수 있는가?"**를 정의한 규칙이 바로 **일관성 모델**입니다. ### 문제의 근원: 8~10장의 도전 과제 * **네트워크(8장):** 패킷은 유실되거나 순서가 뒤바뀝니다. * **장애 감지(9장):** 노드가 죽은 건지, 그냥 느린 건지 알 수 없습니다. * **합의(10장):** 과반수(Quorum)의 동의 없이는 무엇도 '진실'이 되지 못합니다. --- ## 1. 복제의 세 가지 아키텍처 복제를 구현하는 방식은 세 가지로 나뉩니다: | 유형 | 설명 | 대표 시스템 | |:---------------------:|:------------------------------------------------|:---------------------------------------------------------------:| | 단일 리더 (Single-Leader) | 모든 쓰기가 리더를 거치며, 리더가 로그를 복제본에 전달. 읽기는 복제본에서도 가능. | MySQL (Async/Semi-sync), Redis Cluster, PostgreSQL, etcd (Raft) | | 다중 리더 (Multi-Leader) | 여러 리더가 쓰기를 받아 서로 동기화. 충돌 해소 필요. | MySQL Group Replication, Cassandra (특정 구성) | | 리더 없음 (Leaderless) | 클라이언트가 여러 노드에 직접 쓰기. 읽기 시 여러 노드에서 값을 비교. | Cassandra, DynamoDB, Riak | **특징:** - 단일 리더는 구현이 가장 단순하고 일관성 보장이 쉽지만, 리더 장애 시 쓰기 중단 시간이 발생합니다. - 다중 리더는 높은 가용성을 제공하지만 충돌 해소가 복잡합니다. - 리더 없음은 높은 가용성을 제공하지만, 최종적 일관성(Eventual Consistency)을 기본으로 합니다. --- ## 2. 일관성 모델의 스펙트럼 일관성은 '된다/안 된다'의 이분법이 아니라, **성능과 정확성 사이의 선택(Trade-off)**입니다. ```mermaid graph LR A["🔴 선형성

마치 단일 노드 DB를 쓰는 것 같은 직관성을 제공 - 클라이언트가 쓰기 후 바로 읽으면 그 값을 읽을 수 있음 **단점:** - 네트워크 지연에 극도로 취약하며 가용성(Availability)이 떨어짐 - 성능이 가장 낮음 **사용 사례:** etcd, ZooKeeper, 단일 마스터 데이터베이스, 금융 거래 --- ### ❷ 순차적 일관성 (Sequential Consistency) **정의:** 모든 연산이 전역적으로 하나의 순서로 나타나지만, 실제 실시간과는 맞지 않을 수 있음. **특징:** - 모든 노드는 동일한 순서를 봄 - 시간 순서는 보장하지 않음 **예시:** ``` 가능: read → write 순서로 해석되어도 괜찮음 ``` **사용 사례:** 분산 메모리 시스템, 일부 멀티프로세서 캐시 일관성 --- ### ❸ 인과적 일관성 (Causal Consistency) – "현실적인 타협안" **정의:** '원인'과 '결과' 관계가 있는 연산들의 순서만 보장함. **특징:** - 선형성보다 빠르며, 대부분의 서비스에 충분한 수준 - 동시에 발생한 연산 간에는 순서를 강제하지 않음 - 인과 관계를 추적하기 위해 버전 벡터(Version Vector) 필요 **예시:** ``` A: 글 작성 (원인) B: 댓글 작성 (결과) ``` 👉 댓글이 먼저 보이면 안 됨, 글이 먼저 보여야 함 **장점:** 선형성보다 성능 우수, 분산 환경에서 자연스러운 인과 관계 모델링 **사용 사례:** 분산 버전 관리, 일부 NoSQL (MongoDB, Cassandra with causal reads) --- ### ❹ 최종적 일관성 (Eventual Consistency) – "가용성 올인" **정의:** 새로운 업데이트가 없다면, 시간이 흐른 뒤 모든 복제본이 동일해짐. **특징:** - 쓰기 직후 읽기는 옛날 값을 볼 수 있음 - 시스템은 멈추지 않고 계속 돌아감 **장점:** - 가용성이 가장 높음 - 네트워크 파티션에 강함 **단점:** - 애플리케이션 수준에서 충돌 해소 필요 **사용 사례:** DNS, Cassandra (기본), Amazon DynamoDB, SNS 피드 --- ## 3. 선형성과 쿼럼: 읽기/쓰기 정족수 단일 리더 시스템에서 선형성을 보장하려면, **쓰기와 읽기 모두 과반수(Quorum) 이상의 노드와 통신**해야 합니다. ### 쿼럼 조건 - **쓰기 쿼럼 (Write Quorum):** `W > N/2` (예: 3노드 → W=2) - **읽기 쿼럼 (Read Quorum):** `R > N/2` **선형적 읽기 조건:** `W + R > N` 이 조건은 읽기 작업이 **쓰기 쿼럼과 교집합을 갖는 최소한 하나의 노드에서 최신 값을 읽도록 보장**합니다. ### 실제 예: Elasticsearch 8 ```json PUT /my-index/_doc/1?consistency=quorum { "title": "Hello World" } ``` - `quorum`은 `(primary + replicas) // 2 + 1`로 해석됨 - 복제본이 과반수 이상에 쓰기 완료되어야 성공 응답을 반환 - 이를 통해 선형성에 가까운 보장을 얻을 수 있음 --- ## 4. 실제 시스템에서의 선택 ### 시스템별 일관성 모델 | 시스템 | 복제 방식 | 기본 일관성 | 선형성 지원 여부 | |:-------------:|:------------------------:|:---------------------|:--------------------------------------| | etcd | Raft (단일 리더) | 선형성 (Linearizable) | 기본 (읽기 쿼럼 필요) | | Redis | Cluster (단일 리더) | 비동기 복제 → 최종적 | WAIT 명령으로 강제 복제 가능 | | Cassandra | 리더 없음 (Leaderless) | 최종적 일관성 | LOCAL_QUORUM/EACH_QUORUM으로 선형성 근접 가능 | | Elasticsearch | 단일 리더 (샤드별 주 복제본) | 기본적으로 실시간 읽기 | consistency=quorum으로 쓰기 선형성 보장 가능 | ### Redis Cluster 예시: WAIT로 선형성 흉내내기 Redis Cluster는 기본적으로 마스터에 쓰고, **비동기적으로 슬레이브에 복제**합니다. 선형성을 강제하려면 `WAIT` 명령을 사용합니다: ```bash SET user_1 "Gildong" WAIT 1 5000 ``` **효과:** - `WAIT`는 쓰기 후 읽기 시점에 선형적 보장을 만들어 줌 - 단, 성능 저하가 발생할 수 있음 ### Cassandra 예시: QUORUM 수준 일관성 Cassandra는 **리더 없는 복제를 사용**하며, 읽기/쓰기 일관성 수준을 각각 설정할 수 있습니다: ```cql CONSISTENCY QUORUM; INSERT INTO users (id, name) VALUES (1, 'Alice'); CONSISTENCY QUORUM; SELECT * FROM users WHERE id = 1; ``` **작동 방식:** - QUORUM은 `(복제 계수 / 2) + 1` 이상의 노드 응답을 요구 - 이를 통해 선형성에 가까운 보장을 얻을 수 있음 - 네트워크 파티션 시에도 쿼럼 조건에 따라 가용성과 일관성 사이의 균형 조절 --- ## 5. CAP 정리와 PACELC ### CAP 정리 CAP 정리는 분산 시스템이 다음 세 가지 중 **두 가지만 완전히 보장**할 수 있다고 말합니다: - **C (Consistency):** 일관성 - **A (Availability):** 가용성 - **P (Partition Tolerance):** 파티션 내성 **시스템 분류:** - **CP 시스템:** 네트워크 파티션 발생 시 일관성을 유지하기 위해 일부 노드의 서비스를 중단합니다. - 예: etcd, ZooKeeper, HBase - **AP 시스템:** 네트워크 파티션 발생 시 가용성을 유지하지만, 일관성이 일시적으로 깨질 수 있습니다. - 예: Cassandra (기본 설정), DynamoDB (최종적 일관성) - **CA 시스템:** 사실상 네트워크 파티션을 고려하지 않는 단일 데이터센터 시스템 (분산 시스템에서는 드뭄) ### PACELC: 현실적인 선택 CAP는 파티션 상황만 다룬다는 한계가 있습니다. **PACELC**는 정상 상황(E, Else)도 고려합니다: ``` if Partition: C (일관성) vs A (가용성) else: L (지연 시간) vs C (일관성) ``` **즉:** 장애가 없을 때조차 우리는 **성능(Latency)을 위해 일관성을 포기하곤 한다**. --- ## 6. 실제 애플리케이션에서의 트레이드오프 실제 애플리케이션은 요구사항에 따라 일관성 모델을 선택합니다: | 요구사항 | 추천 일관성 | 이유 | |:-------------------|:----------:|:---------------------------------| | 금융 거래, 계정 잔액 | 선형성 | 데이터 손실이나 중복 집계를 막아야 함 | | 소셜 미디어 피드, 댓글 | 인과적 일관성 | 좋아요 → 댓글 같은 인과 관계만 보장해도 충분 | | 제품 카탈로그, DNS | 최종적 일관성 | 일시적 불일치가 비즈니스에 치명적이지 않음 | | SNS 좋아요 수, 조회수 | 최종적 일관성 | 정확성보다 가용성이 중요 | --- ## 7. 핵심 요약 및 생각해 볼 점 ### 핵심 요약 ✓ **일관성은 이진적 선택이 아닌 스펙트럼**이다. ✓ **선형성은 정확하지만 느리고**, 최종적 일관성은 **빠르지만 부정확**할 수 있다. ✓ **쿼럼(Quorum) 설정**을 통해 개발자가 직접 일관성 수준을 조정할 수 있다. ✓ **실제 시스템은 원하는 일관성을 선택적으로 적용**하도록 설계되어 있다. ### 생각해 볼 문제 **Q1.** 은행 송금 시스템을 설계한다면, CAP 중 무엇을 포기해야 할까요? (힌트: 돈이 복사되면 안 됩니다.) **Q2.** Cassandra에서 `W + R > N` (쿼럼 조건)을 만족하면 항상 선형성이 보장될까요? 왜 아닐 수도 있을까요? **Q3.** SNS의 '좋아요' 수가 모든 사람에게 실시간으로 똑같이 보이지 않아도 되는 이유는 어떤 일관성 모델 때문일까요? --- ## 마무리 이제 11장의 여정이 완성되었습니다: - **8장(네트워크):** 신뢰할 수 없는 통신 채널 - **9장(장애 감지):** 불완전한 실패 감지 - **10장(합의):** 과반수 기반의 의사결정 - **11장(일관성):** 그 모든 것들의 결과로 선택할 수 있는 일관성 모델들 **복제는 가용성과 확장성을 위한 필수 기술**이지만, **일관성과의 트레이드오프**가 존재합니다. 시스템 설계자로서 여러분의 역할은 비즈니스 요구사항에 맞는 일관성 모델을 선택하는 것입니다.
