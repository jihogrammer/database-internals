분산 시스템의 일관성: 안티-엔트로피에서 합의까지

Chapter 12-13-14 • 작성일: April 1, 2026
Database Internals 12–14장 정리

분산 시스템에서 데이터를 여러 노드에 복제하면 가용성과 확장성이라는 큰 장점을 얻을 수 있습니다. 하지만 그 대가로 복제본 간 데이터 불일치라는 숙명적인 문제를 안게 됩니다.

이 글에서는 분산 시스템이 복제본 간 일관성을 유지하기 위해 사용하는 세 가지 계층적 메커니즘을 살펴봅니다:

  1. 안티-엔트로피 (Anti-Entropy) : 시간이 지나면서 벌어진 불일치를 점진적으로 수렴
  2. 분산 트랜잭션 (Distributed Transactions) : 여러 노드에 걸친 작업의 원자성 보장
  3. 합의 (Consensus) : 분산 환경에서 단일 결정에 대한 합의 도출

1. 안티-엔트로피: 결과적 일관성의 기반

"무질서도(entropy)가 증가하는 것을 막는 자기 치유 메커니즘"

안티-엔트로피는 복제본 간 데이터 불일치를 주기적으로 탐지하고 동기화하는 백그라운드 프로세스입니다.

1.1 왜 필요한가?

분산 시스템에서는 다양한 이유로 복제본 간 차이가 발생합니다:

  • 네트워크 파티션으로 일부 노드가 업데이트를 놓침
  • 노드 장애 후 복구 시점에 최신 데이터가 없음
  • 메시지 유실이나 중복 전송

안티-엔트로피가 없다면, 이러한 차이는 영원히 해소되지 않을 수 있습니다.

1.2 머클 트리 (Merkle Tree): 효율적 불일치 탐지

머클 트리는 전체 데이터를 비교하지 않고 변경된 부분만 빠르게 찾아내는 자료구조입니다.

graph TD
subgraph MerkleTree [머클 트리 구조]
R[Root Hash = hash / H1 + H2]
H1[Hash = hash / A + B]
H2[Hash = hash / C + D]
A[Leaf: Data A]
B[Leaf: Data B]
C[Leaf: Data C]
D[Leaf: Data D]

R --> H1
R --> H2
H1 --> A
H1 --> B
H2 --> C
H2 --> D
end

동작 방식:

  1. 각 노드는 자신의 데이터를 머클 트리로 구성
  2. 두 노드가 루트 해시만 비교 → 같으면 전체 데이터 일치
  3. 다르면 자식 노드로 내려가며 불일치 범위를 좁힘
  4. 실제로 다른 부분만 전송하여 동기화

효과: 전체 데이터를 비교하는 대신 O(log N) 복잡도로 불일치 탐지 가능.

1.3 읽기 복구 (Read Repair): 핫 데이터의 빠른 수렴

읽기 복구는 읽기 요청 시점에 불일치를 즉시 해소하는 기법입니다.

sequenceDiagram
    participant Client
    participant Coordinator
    participant Replica1 (stale)
    participant Replica2 (latest)
    Client ->> Coordinator: read(key)
    Coordinator ->> Replica1: digest request
    Coordinator ->> Replica2: digest request
    Replica1 -->> Coordinator: digest (vclock: {A:3})
    Replica2 -->> Coordinator: digest (vclock: {A:5})
    Note over Coordinator: 불일치 발견 (Replica2가 최신)
    Coordinator -->> Client: Replica2의 데이터 반환 (응답 완료)
    Note over Coordinator, Replica1: 백그라운드에서 stale replica 업데이트
    Coordinator ->> Replica1: write 최신 데이터

특징:

  • 다이제스트 읽기(Digest Read): 해시, 벡터 클락 등 작은 메타 정보만 먼저 교환
  • 클라이언트 응답 후 비동기 복구: 읽기 지연에 영향 최소화
  • 자주 읽히는 데이터는 빠르게 일관성 수렴

1.4 가십 프로토콜 (Gossip Protocol): 멤버십과 상태 전파

가십은 노드들이 주기적으로 무작위로 다른 노드와 정보를 교환하며 점진적으로 전체에 전파하는 방식입니다.

특징설명
Push-Pull내 정보를 보내고(Push) 상대방 정보를 받아옴(Pull)
팬아웃(Fanout)한 라운드에 접촉하는 노드 수 (속도 vs 부하 트레이드오프)
부분 뷰(Partial View)전체가 아닌 일부 노드만 연결하여 확장성 확보
수렴 속도O(log N) 라운드로 전체 노드 도달

실제 사례:

  • Cassandra: 가십으로 멤버십 관리, 안티-엔트로피로 데이터 일치
  • Elasticsearch: 노드 검출(discovery)에 유사한 메커니즘 사용

1.5 힌티드 핸드오프 (Hinted Handoff): 장애 중 쓰기 가용성

sequenceDiagram
    participant Client
    participant Coordinator (A)
    participant ReplicaB (down)
    participant HintNode (C)
    Client ->> Coordinator: write
    Coordinator ->> ReplicaB: fail
    Coordinator ->> HintNode: hinted handoff (임시 저장)
    HintNode -->> Coordinator: ack
    Coordinator -->> Client: success
    Note over ReplicaB: 복구 후
    HintNode ->> ReplicaB: deliver hint

핵심: 장애 노드가 있어도 임시 저장 후 복구 시점에 전달 → 쓰기 가용성 극대화

2. 분산 트랜잭션: 원자성의 확장

단일 노드에서는 ACID 트랜잭션이 익숙합니다. 분산 트랜잭션은 여러 노드에 걸친 작업을 하나의 원자적 단위로 묶는 기술입니다.

2.1 Calvin: "미리 순서를 정하자"

Calvin은 2012년 Yale University에서 발표한 분산 트랜잭션 프로토콜로, 실행 전에 트랜잭션 순서를 합의하는 독특한 접근법을 취합니다.

flowchart TD
    subgraph Sequencing [Sequencing Layer]
        C[Client Request] --> S[Sequencer]
        S --> L[Global Transaction Log]
        L --> R1[Replica 1]
        L --> R2[Replica 2]
    end

    subgraph Execution [결정적 실행]
        R1 --> Sch[Scheduler]
        Sch --> Lock[Strict 2PL]
        Lock --> Store[(Storage)]
    end

핵심 원리:

  1. 모든 트랜잭션은 sequencer를 통해 글로벌 로그에 순서가 합의됨
  2. 모든 replica가 동일한 순서로 트랜잭션 실행
  3. 순서가 정해졌으므로 2PC 없이도 분산 트랜잭션 처리 가능

장점:

  • 2PC 불필요 → 성능 향상
  • 클록 스큐 영향 없음
  • 이론적 정확성 보장

단점:

  • 읽기 트랜잭션도 sequencer 필요
  • "SELECT 후 결과에 따라 UPDATE" 같은 종속 트랜잭션 미지원

실제 구현: FaunaDB가 Calvin 프로토콜 기반

2.2 Spanner: "TrueTime으로 정렬하자"

Google Spanner는 GPS + 원자시계 기반 TrueTime API를 사용하여 글로벌 분산 트랜잭션을 구현합니다.

flowchart LR
    subgraph TrueTime [TrueTime API]
        TT[GPS + Atomic Clock]
        TT -->|bound < 7ms| Interval[시간 구간 반환]
    end

subgraph Transaction [트랜잭션 처리]
Write -->|commit_ts = now.latest|Timestamp
Timestamp -->|wait until now.earliest > timestamp|CommitWait
CommitWait --> Release[락 해제]
end

TrueTime API:

  • tt.now(): [earliest, latest] 구간 반환 (불확실성 7ms 이내)
  • tt.after(t): t가 확실히 지났는지 확인

트랜잭션 처리:

  1. 쓰기 트랜잭션은 2PC + Paxos로 처리
  2. 커밋 타임스탬프 = now().latest
  3. Commit Wait: 해당 타임스탬프가 확실히 지날 때까지 대기 → 외부 일관성 보장

장점:

  • 읽기 전용 트랜잭션은 락 없이 MVCC로 처리
  • SQL 완전 지원 (종속 트랜잭션 가능)
  • 지리적 분산에 최적화

단점:

  • GPS + 원자시계라는 하드웨어 인프라 필요
  • 다중 샤드 트랜잭션 시 2PC 오버헤드

실제 구현: Google Cloud Spanner, CockroachDB (HLC로 유사 구현)

2.3 Percolator: BigTable 위의 트랜잭션

Percolator는 Google이 BigTable 위에 다중 행 트랜잭션을 구현하기 위해 개발한 시스템입니다. TiKV와 같은 현대 분산 DB의 기초가 되었습니다.

sequenceDiagram
    participant Client
    participant TSO as Timestamp Oracle
    participant BigTable
    Note over Client, BigTable: Prewrite Phase
    Client ->> TSO: get start_ts
    TSO -->> Client: start_ts
    Client ->> BigTable: write data + lock (primary)
    Client ->> BigTable: write data + lock (secondary)
    Note over Client, BigTable: Commit Phase
    Client ->> TSO: get commit_ts
    TSO -->> Client: commit_ts
    Client ->> BigTable: remove primary lock + write record
    Client ->> BigTable: remove secondary locks

핵심 구조: 각 Column을 3개의 내부 Column으로 분리

내부 Column용도
c:lock트랜잭션 락 정보 (primary lock이 원자성 보장)
c:write커밋된 데이터의 위치 (commit_ts → start_ts 매핑)
c:data실제 사용자 데이터 (start_ts 기준)

Timestamp Oracle (TSO):

  • 단일 머신에서 초당 200만 개 타임스탬프 발급 가능
  • 모든 읽기/쓰기가 TSO에서 timestamp를 받아 수행

충돌 처리:

  • 읽기: lock 발견 시 backoff 후 재시도
  • 쓰기: prewrite 단계에서 lock 존재 시 abort
  • 크래시 복구: primary lock 상태로 commit/rollback 결정

3. 합의 (Consensus): 분산 시스템의 결정 엔진

합의는 여러 노드가 하나의 값에 대해 "합의"하는 문제로, 분산 시스템의 가장 근본적인 메커니즘입니다.

3.1 합의가 필요한 이유

문제설명
리더 선출단일 장애점(SPOF) 제거
데이터 복제여러 노드에 동일한 데이터 유지
분산 트랜잭션다중 노드 트랜잭션의 원자성 보장
구성 변경클러스터에 노드 추가/제거 시 안전 처리

3.3 Raft 상세: "이해 가능한 합의 알고리즘"

Raft는 2014년 Diego Ongaro와 John Ousterhout가 발표한 논문에서 "이해할 수 있는 합의 알고리즘" 을 목표로 설계되었습니다.

리더 선출 (Leader Election)

stateDiagram-v2
    [*] --> Follower
    Follower --> Candidate: election timeout
    Candidate --> Leader: votes from majority
    Leader --> Follower: discovers higher term
    Candidate --> Follower: discovers higher term
    Leader --> Leader: heartbeat
  • 모든 노드는 팔로워(Follower) 로 시작
  • election timeout(랜덤 150-300ms) 동안 하트비트 없으면 후보자(Candidate) 로 전환
  • 과반수(majority) 득표 시 리더(Leader) 로 전환

로그 복제 (Log Replication)

sequenceDiagram
    participant Client
    participant Leader
    participant Follower
    Client ->> Leader: command
    Leader ->> Leader: append to log
    Leader ->> Follower: AppendEntries RPC
    Follower -->> Leader: success
    Note over Leader: majority success → commit
    Leader -->> Client: response
    Leader ->> Follower: apply commit

Raft의 차별점:

  • 리더 기반: 모든 요청은 리더를 통해 처리
  • 로그 구조: 명시적 로그 정렬과 커밋 인덱스 관리
  • 구성 변경: 공동 합의(joint consensus)라는 명확한 메커니즘

3.4 합의의 한계: FLP 불가능성

"비동기 분산 시스템에서 단일 장애가 발생할 가능성이 있는 경우, 합의는 무한히 지연될 수 있다"

실제 극복 방법:

  • 부분 동기 모델(Partial Synchrony) 가정: 대부분의 시간에는 동기적, 일시적으로 비동기적
  • 타임아웃, 하트비트로 "실질적" 합의 보장

마치며

분산 시스템에서 일관성을 유지하는 일은 단일 노드에서의 ACID보다 훨씬 복잡한 문제입니다. 안티-엔트로피는 느리지만 꾸준한 수렴을, 분산 트랜잭션은 원자적 보장을, 합의는 결정의 불변성을 제공합니다.

각 시스템은 자신의 목적(OLTP vs OLAP, 강한 일관성 vs 가용성)에 따라 이 기술들을 선택적으로 조합합니다. Cassandra는 AP 시스템으로 안티-엔트로피와 가십에 집중하고, Spanner는 CP 시스템으로 합의와 TrueTime을 활용하며, Elasticsearch는 검색 최적화를 위해 분산 트랜잭션을 포기하는 대신 다른 메커니즘으로 대체합니다.

이러한 기술들을 이해하는 것은 분산 시스템의 트레이드오프를 설계에 반영하는 첫걸음입니다.
